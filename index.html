1: How to define a measure

DON'T

- Don't use sdmx:obsValue as a measure

DON'T

- Don't define measures as sub-properties of sdmx:obsValue

DO

OPEN ISSUES

- What URI to use for measures. Indicators Interoperability (name, calculation formula, data etc.)

2: Define cubes with multiple measures?

OPEN ISSUES

- Should a cube have one or multiple measures?
- Use cubes with multiple measures only when measures are closely related to a single observational event

3: How to define a unit

DO

- Always define unit for measures

DO

- QUDT vocabulary should be preferred for the units of measure

4: How to define multiple units per measure

Scenario: Publish data about unemployment with two units (percent, count)

OPTION 1 (Publish multiple cubes with one unit)

- Publish two cubes about unemployment with a single unit. :cube1 with qudt:Percent and :cube2 with qudt:Number
- Define the unit at the qb:MeasureProperty

OPTION 2 (Publish one cube with multiple units)

- Publish one cube about unemployment with two units (qudt:Percent, qudt:Number)
- Define the unit at each qb:Observation

5: How to define multiple measures with multiple units

Scenario: Publish data about unemployment (percent, count) and poverty (count)

OPTION 1 (Multi-measure observations)

- Use multiple measures at one observation. For example :obs1 has two measures :unemployment and :poverty

OPTION 2 (Measure dimension)

- Use observations with a single measure
- Use the dimension qb:MeasureType to denote the measure used by the observation

6: How to define dimensions values

DO

- Use a code list to indicate the set of potential dimension values
- Use rdfs:range to define the type of dimension values
*Exception: a code list should not be defined when the rdfs:range is a data-type e.g. xsd:date, xsd:integer, xsd:boolean

7: How to define code lists

DON'T

- Don’t include irrelevant values under a single dimension

DO

- Define a separate dimension for each descriptive category of the data (e.g. sex, age)
- Use a separate code list for each dimension

8: How to define hierarchical code list

DON'T

- Don’t define hierarchical data at the same level

DO

- Use hierarchies at hierarchical data

HOW (OPTION 1 SKOS)

- SKOS can define broader/narrower relations

HOW (OPTION 2 QB)

- QB can define new broader/narrower relations e.g. within
- QB can express more relations not covered by SKOS or XKOS e.g. administeredBy vs within

HOW (OPTION 3 XKOS)

- XKOS can define broader/narrower relations
- XKOS can define hierarchical levels e.g. country, city

9: How to define “total” values?

DO

Define total values for the dimensions (where applicable)

DON'T

Don’t define total values at the same hierarchical level with the rest dimension values

HOW (OPTION 1)

- Define total values at the top of a hierarchy (see Rec 6)

HOW (OPTION 2)

- Total values can be defined at slices. The total value for a dimension is represented by leaving out this dimension from the slice.
-The example shows a slice that defines the total unemployment (35.9%) for all the people (male and female) with age 15-24, at Brussels the year 2016

11: How to define “sex” dimension?

DO

- Re-use the dimension sdmx:sex
* The codelist sdmx-code:sex has to be updated according to Rec 5 i.e. sdmx-code:sex-T should not be at the same hierarchical level with sdmx-code:sex-M and sdmx-code:sex-F
* If more notions of sex are needed (e.g. hermaphroditism, transgender), then define a new dimension and/or code list

12: How to define “geo” and “age” dimension?

DO

- Define new dimensions for geography and age by extending sdmx:refArea and sdmx:age respectively
- Associate the new dimensions with a code list

13: Dimension with Single Value

DO

- If a dataset has a single value for a dimension, then define this value at each observation e.g. 2016 census data
